# Copilot custom instructions — BRD reverse-engineering pipeline (v2)

Append this to `.github/copilot-instructions.md` (or keep as a separate
section if you already have instructions from an earlier pipeline in this
repo — agent names in this set are prefixed distinctly and won't collide).

This repo uses an 8-step, 7-agent BRD reverse-engineering pipeline, based
on: Plan → Discover → Interpret → Extract → Draft → Review → Trace →
Validate. All agents in this pipeline share these rules.

## Confidence labeling (mandatory on every requirement)

Every requirement, rule, or workflow description produced anywhere in this
pipeline must be tagged with exactly one of:
- **Confirmed** — directly supported by code, UI, database schema, API
  contract, tests, or existing documentation. Cite the specific file,
  class, method, or document.
- **Inferred** — derived from multiple technical observations that
  together imply a behavior, but no single source states it directly.
  Explain the inference chain briefly.
- **Assumed** — a reasonable guess where evidence is thin; requires
  stakeholder confirmation before being treated as final.
- **Unresolved** — insufficient evidence exists to say anything reliable;
  state this plainly rather than guessing.

Never state a requirement without one of these four labels. Never upgrade
an Assumed or Unresolved item to Confirmed without new evidence.

## BRD document structure (used by the Authoring Agent)

This structure is grounded in two reference templates provided by the
business analyst: a detailed BRD/SRS-hybrid template (front matter,
System Features with a formal use-case schema, external interfaces,
nonfunctional requirements, appendices) and a simpler business-facing BRD
sample (Executive Summary, Business Objectives, Stakeholders table,
Success Criteria, Approval sign-off). Use this merged structure for every
BRD this pipeline produces:

**Front matter**
- Document Version and Date
- Authors — for a reverse-engineered BRD, list this as generated by the
  agent pipeline (name the run ID), NOT invented human names
- Document History
- Approvals — leave this table BLANK (Stakeholder / Title / Signature /
  Date columns, no rows filled in). Sign-off is a human action; no agent
  may populate names or signatures here.

**1. Introduction**
- 1.1 Purpose of the Document
- 1.2 Document Conventions, Terms, and Definitions — must define
  Confirmed/Inferred/Assumed/Unresolved here, since that is this
  pipeline's core convention
- 1.3 Intended Audience and Reading Suggestions
- 1.4 Scope — 1.4.1 Project Scope (In Scope / Out of Scope), 1.4.2 BRD Scope
- 1.5 References (repo URL, run ID, any existing docs read)

**2. Overall Description**
- 2.1 Executive Summary
- 2.2 Business Background / Current-State Overview
- 2.3 Business Objectives
- 2.4 Product Features (high-level summary; detail lives in Section 3)
- 2.5 Stakeholders and User Classes — merge the PDF template's
  Role/Department/Responsibilities stakeholder table with the actual
  end-user roles identified from the application
- 2.6 Operating Environment (tech stack — Confirmed, cited to config/build
  files)
- 2.7 Design and Implementation Constraints
- 2.8 Assumptions, Dependencies, and Risks

**3. System Features** — one numbered subsection per feature/requirement
identified. For each feature, include:
- 3.N.1 Description
- 3.N.2 Business Rules
- 3.N.3 Functional Requirements (FR-###)
- 3.N.4 Use Case(s), using this EXACT schema for every use case: Use Case
  ID, Use Case Name, Actors, Description, Trigger, Preconditions,
  Postconditions, Normal Flow, Alternative Flow, Exceptions, Includes,
  Priority, Frequency of Use, Business Rules, Special Requirements,
  Assumptions, Notes and Issues.

Every requirement and use-case field must carry a confidence label
(Confirmed/Inferred/Assumed/Unresolved) and, where Confirmed, a citation
to the specific file/class/method.

**4. External Interface Requirements**
- 4.1 User Interface
- 4.2 Hardware Interfaces
- 4.3 Software Interfaces (APIs, integrations)
- 4.4 Communications Interfaces
- 4.5 Reporting Requirements

**5. Other Nonfunctional Requirements**
- 5.1 Performance, 5.2 Security, 5.3 Data Requirements, 5.4 Software
  Quality Attributes

**6. Other Requirements**

**7. Success Criteria** — from the PDF template. Be honest here: a
reverse-engineered BRD usually cannot derive real target metrics (e.g.,
"reduce tickets by 50%") from code alone. Default every Success Criteria
item to Assumed or Unresolved unless the codebase contains an actual
stated SLA/target (e.g., in config or a comment) that supports Confirmed.

**8. Approval and Sign-off** — blank table, per Front Matter rule above.

**Appendix A: Glossary**
**Appendix B: Analysis Models** — diagrams (context diagram, swim lanes,
data flow). This pipeline does not generate diagrams unless a diagramming
tool is explicitly added — mark this appendix Unresolved/Not Generated
rather than describing a diagram in prose as if one exists.
**Appendix C: Issues List** — every Assumed/Unresolved item and open
question surfaced anywhere in the document, consolidated here.
**Appendix D: Screenshots, Mock-Ups** — Unresolved/Not Generated unless
actual UI screenshots were captured during discovery.
**Appendix E: Test Scenarios** — map directly to actual test files/methods
found during discovery; cite them.
**Appendix F: Traceability Matrix** — filled in by the Traceability Agent
(not part of either reference template, but required by this pipeline).

## Execution log and intermediate drafts (mandatory — required for eval)

The orchestrator creates one run folder per pipeline execution at:
`docs/brd_runs/<run-id>/` where `<run-id>` is a UTC timestamp like
`2026-07-15T14-30-00Z`.

Every agent in this pipeline MUST, using its file-write tool:
1. On starting, APPEND a row to `docs/brd_runs/<run-id>/00_execution_log.md`:
   `| <UTC timestamp> | <agent-name> | <model> | STARTED | — |`
2. On finishing, save its full output to its own numbered file in the same
   folder (see file names in each agent's own instructions below), then
   APPEND a completion row:
   `| <UTC timestamp> | <agent-name> | <model> | COMPLETED | output ~<N> words; <one-line summary> |`
3. If an agent cannot complete its task (missing tool access, no data
   found, etc.), it still APPENDS a row with status `FAILED` or
   `PARTIAL` and a one-line reason — never silently skip logging.

The log file is append-only. No agent overwrites earlier rows. This log,
plus every numbered draft file, is the eval artifact for this pipeline —
treat writing it as part of the task, not optional bookkeeping.

## Swim-lane diagram generation (Step 9, optional)

A ninth, optional step exists for generating executive-ready, editable MS
Visio (.vsdx) swim-lane diagrams — one per use case/process flow in the
validated BRD — handled by `brd-swimlane-diagrams`. This is invoked
separately from the default 8-step run, only when requested.

**Additional grounding sources**, beyond the BRD itself: the 17
microservice `.md` files, the Discovery Deck, meeting minutes, and the
Functional Overview Slides. An element (actor, step, decision gate,
business rule, or integration point) is drawable only if corroborated in
at least one of these sources or the BRD itself — anything else is
excluded from the diagram and logged as Unresolved in a companion
`<use-case-id>.flow.json`, not drawn with a hedge.

**Hard prerequisite:** a base Visio template at
`docs/templates/swimlane-template.vsdx`, containing reusable master
shapes for lane containers, start/end, process, decision, and
integration-call shapes, plus a connector. This must be created once, by
a human, in real Visio (or adapted from one of Visio's built-in
cross-functional flowchart templates) and committed to the repo — the
generation tooling opens and populates this template, it does not create
a blank Visio canvas from nothing. Do not attempt Step 9 without it.

Output convention: `docs/brd_runs/<run-id>/swimlanes/<use-case-id>.vsdx`,
alongside its `<use-case-id>.flow.json` audit trail. Update Appendix B of
the BRD (Analysis Models) to point to these files once generated, rather
than leaving it marked Not Generated, if the user asks for the BRD to be
re-finalized after Step 9 runs.
